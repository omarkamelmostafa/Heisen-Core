# Project Constitution — Binding Architecture Rules

## Project Identity
- Name: New Starter Kit
- Type: Full-stack generic starter kit application
- Stack: Next.js 15 (App Router) + Express.js + MongoDB + Redis
- Auth Model: JWT access tokens + HTTP-only refresh token cookies
- Authorization: RBAC with @casl/ability (Phase 4, Before the final phase 5)
- Testing: Vitest (unit + integration)
- UI: Tailwind CSS + shadcn/ui components

## SECTION 2: Frontend Architecture Rules

### Rule F1: Hook-Driven Feature Architecture
All business logic lives in custom hooks. Pages are thin connectors that import hooks and components, then wire them via props. Components are dumb (props only, no Redux, no API calls, no navigation).
Exceptions: auth-bootstrap.jsx, protected-guard.jsx, and public-guard.jsx are infrastructure components that are intentionally smart. Do not extract their logic into hooks.

### Rule F2: Feature-Based Folder Structure
New features must follow this structure:
- src/features/[feature]/hooks/ for custom hooks
- src/features/[feature]/components/ for feature UI components
- src/components/ui/ for shared dumb components only
- src/app/ for Next.js pages (connectors only)
- src/lib/ for utilities, config, notify
- src/store/ for Redux store and slices
Do not place feature-specific components in src/components/.

### Rule F3: Route Guards Use isBootstrapComplete
PublicGuard and ProtectedGuard check isBootstrapComplete from Redux. They do NOT check isLoading. isLoading changes during form submissions and causes guards to unmount forms. isBootstrapComplete is set once after AuthBootstrap finishes and never changes again.

### Rule F4: No Direct Sonner Imports
Only src/lib/notify.js imports from "sonner". Every other file uses import { notify } from "@/lib/notify". Toast calls use explicit parameter forwarding (id, description, duration, action). No spread operator to Sonner API.

### Rule F5: Toast IDs for Interceptor Errors
All interceptor toasts must use fixed IDs to prevent duplicate stacking: session-expired, global-403, global-429, global-500, global-5xx, global-network, cross-tab-logout.

### Rule F6: AuthBootstrap Uses Raw Fetch
AuthBootstrap calls the refresh endpoint using raw fetch(), NOT the interceptor-equipped axios instance. This prevents circular dependency. It uses a useRef guard (hasAttemptedRestore) to prevent double execution in React Strict Mode.

### Rule F7: BroadcastChannel Rules
- Channel name: auth_channel
- LOGOUT broadcast: all tabs clear state and redirect. Uses sessionStorage flag logout_source to prevent same-tab duplicate toast. 1500ms setTimeout with pathname check before redirect.
- LOGIN broadcast: other tabs do window.location.href full reload to /dashboard. Uses sessionStorage flag login_source to prevent same-tab duplicate reload.
- Stale flags (logout_source, login_source) cleared on AuthBootstrap mount.
- Never broadcast during AuthBootstrap silent failure.

### Rule F8 (Future Standard): New Pages Must Follow This Template
Every new page must:
1. Be wrapped in ProtectedGuard or PublicGuard as appropriate
2. Import a custom hook for all business logic
3. Import components for UI
4. Wire hook return values to component props
5. Contain only local UI state (show/hide toggles, form visibility) 
6. Never import useDispatch or useSelector directly

### Rule F9 (Future Standard): Strict Error Boundary Requirement for New Routes
Any future route group created in this project (e.g., `src/app/(settings)/`) MUST implement a dedicated `error.jsx` file. This is a mandatory architectural standard for all upcoming UI development to ensure graceful degradation.
Every new `error.jsx` must strictly adhere to this contract:
- Must include the `"use client"` directive at the top.
- Must accept `error` and `reset` as props.
- Must log the error silently via a `useEffect` hook.
- Must render a user-friendly fallback UI (absolutely NO raw stack traces exposed in production).
- Must include a functional "Try Again" button that triggers the `reset()` function.
- Must be styled consistently with the existing global error boundaries.

## SECTION 3: Backend Architecture Rules

### Rule B1: App/Server Separation
- app.js: Creates Express app, registers middleware, mounts routes, exports app. Zero side effects on import. No dotenv, no DB connection, no app.listen().
- index.js: Imports app, calls dotenv.config(), validateEnv(), connectToMongo(), app.listen(). Production entry point only.
- Tests import app.js directly.

### Rule B2: Error Response Format
All error responses use this envelope:
success: false, message: "Human-readable error", errorCode: "MACHINE_READABLE_CODE", timestamp: "ISO-8601"
The field name is errorCode (not code, not error_code).
Standard codes: TOKEN_EXPIRED, TOKEN_INVALID, NO_ACCESS_TOKEN, MISSING_REFRESH_TOKEN, RATE_LIMITED, ALREADY_VERIFIED, VERIFICATION_TOKEN_EXPIRED.

### Rule B3: Cookie Configuration
- Cookie name: refresh_token (underscore, not camelCase)
- Path: / (not scoped to refresh endpoint)
- HttpOnly: always true (never false, regardless of NODE_ENV)
- Secure: true in production, false in development
- SameSite: strict
- MaxAge: 7 days (must match refresh token TTL)

### Rule B4: Environment Validation
- Critical variables (JWT secrets): throw and stop server
- Recommended variables (Mailtrap, CORS): warn and continue
- validateEnv() runs BEFORE any middleware or DB connection
- Tests set their own env vars BEFORE importing app.js

### Rule B5: Trust Proxy
app.set('trust proxy', 1) is the first line after Express app creation. Required for correct IP-based rate limiting behind proxies.

### Rule B6: Rate Limiter Test Bypass
Rate limiters bypass in test environment via the factory function. Applied at factory level so ALL limiters are bypassed. NODE_ENV === 'test' check.

## SECTION 4: Validation Rules

### Rule V1: Shared Helpers
- passwordRules(field) for all password fields except login
- emailRules({ checkDisposable }) for all email fields
- Login does NOT validate password strength (only existence)
- Forgot-password does NOT check disposable emails

### Rule V2: Validation Chain Patterns
- .bail() after every .notEmpty() and .isString()
- .customSanitizer() on OTP token fields (converts number to string)
- No .escape() anywhere (XSS handled at output/rendering layer)
- Error messages are user-facing, not developer-facing

### Rule V3: Verification Token Field Name
The verify-email endpoint expects { token } in req.body.
Not code, not verificationCode, not otp.

## SECTION 5: Testing Rules

### Rule T1: Vitest only. Not Jest.

### Rule T2: vi.mock() is hoisted. Never mock inside beforeEach with dynamic imports. Modules are already loaded by import time.

### Rule T3: Never mock bcrypt, jsonwebtoken, or Node.js crypto. Mock only external I/O (email service, Cloudinary).

### Rule T4: Integration Test Patterns
- Integration tests use supertest + mongodb-memory-server
- request(app) for stateless single-request tests
- request.agent(app) for cookie-dependent multi-step flows
- beforeEach wipes all collections + vi.clearAllMocks()
- vi.useRealTimers() must be restored after time-travel tests
- Integration tests must not call app.listen()

### Rule T5: Separate test commands.
npm run test:unit (fast, no DB), npm run test:integration (needs MongoMemoryServer), npm run test:coverage (unit only, 100% thresholds).

### Rule T6: 3-Layer assertions in integration tests.
Every happy-path test asserts HTTP status + response body + database state.

### Rule T7: Coverage thresholds are 100% on all auth utility files.
### Rule T7: Coverage Thresholds
Coverage thresholds are 100% on all auth utility files.
Coverage scope must be explicitly listed in vitest.config.js.
Do not rely on auto-discovery for coverage targets.

### Rule T8: Mock Email Service Default Export, Not Class Constructor
The email service exports a default singleton instance.
Integration tests must mock the default export, not rely on
EmailService.mock.instances. Token extraction in tests must
read mock calls from the mocked singleton methods.

## SECTION 6: Security Rules

### Rule S1: Enumeration Prevention
Login wrong password and non-existent email return identical error messages and status codes. Forgot-password returns identical response regardless of email existence.

### Rule S2: Separate JWT Secrets
ACCESS_TOKEN_SECRET and REFRESH_TOKEN_SECRET must be different.

### Rule S3: Access tokens in Redux memory only. Never localStorage.

## SECTION 7: Unverified Behaviors
These behaviors are implemented in the codebase but NOT explicitly tested. They should be verified before being promoted to binding rules:

1. Token reuse detection: if a revoked refresh token is used, ALL user sessions should be revoked. (Implemented in refresh use case, no dedicated integration test.)

2. Password reset revokes ALL active refresh tokens. (Implemented in reset-password use case, no dedicated integration test verifying all sessions are cleared.)

## SECTION 8: Naming Conventions

### Files:
- Components: kebab-case (login-form.jsx)
- Hooks: camelCase with use prefix (useLogin.js)
- Utils: kebab-case (hash-utils.js)
- Tests: filename.test.js in __tests__/unit/ or integration/
- Models: PascalCase (User.js, RefreshToken.js)

### Code:
- Redux actions: camelCase (setCredentials)
- Error codes: UPPER_SNAKE_CASE (TOKEN_EXPIRED)
- API routes: kebab-case (/verify-email)
- Cookie names: snake_case (refresh_token)
- Environment variables: UPPER_SNAKE_CASE (ACCESS_TOKEN_SECRET)

## SECTION 9: Forbidden Patterns

1. import toast from "sonner" in any file except notify.js
2. useDispatch or useSelector inside a component (except infrastructure)
3. dotenv.config() in app.js
4. app.listen() in app.js
5. Mocking bcrypt or jsonwebtoken in tests
6. Dynamic import for mocking inside beforeEach
7. isLoading in route guard conditions
8. .escape() in validation chains
9. Spread operator in notify.js options
10. Storing access tokens in localStorage
11. Same JWT secret for access and refresh tokens
12. Different error messages for wrong password vs missing email on login
