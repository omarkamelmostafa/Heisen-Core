# FEATURE LIFECYCLE REFERENCE — New Starter Kit

## How To Use This Document

This document describes every module a feature touches from the moment
a user clicks a button to the moment the database writes and the UI
updates. It is derived from a verified contract audit of the full stack.

**For any new feature:**
1. Read this document to understand the flow
2. Identify which stages your feature needs (not all features need all stages)
3. Use the atomic task list at the bottom — each task targets exactly one file
4. Execute tasks in order (backend first, then frontend state, then UI last)
5. After implementation, run the boundary verification checklist

### STAGE 1: Zod Schema (Client-Side Validation Gate)

**File:** `frontend/src/lib/validations/auth-schemas.js`

**Pattern:**
```javascript
export const [featureName]Schema = z.object({
  fieldA: z.string().min(1, "Field A is required"),
  fieldB: sharedValidator,  // reuse existing shared schemas
  fieldC: z.string().min(1, "..."),
})
.refine((data) => /* cross-field check 1 */, {
  message: "...",
  path: ["fieldName"],
})
.refine((data) => /* cross-field check 2 */, {
  message: "...",
  path: ["fieldName"],
});
```

**Shared validators available:**
| Variable | Validates | Used For |
|----------|-----------|----------|
| `emailSchema` | Required + valid email | Any email field |
| `passwordSchema` | Required + 8-128 chars + uppercase + lowercase + number + special | New password fields (NOT login, NOT current password) |
| `loginPasswordSchema` | Required + 8 chars min | Login password only |
| `nameSchema` | Required + 3-16 chars + letters/spaces only | First/last name |

**Rules:**
- Current/old password fields use simple `.min(1)` — no strength validation
- New password fields use `passwordSchema` — full strength validation
- `.refine()` for cross-field checks (password match, not-same-as-old, etc.)
- Field names in schema MUST match backend validation field names exactly

**Output:** After validation, `useForm` produces an object with exact field names matching the schema keys.

### STAGE 2: UI Component (Form + Submit Trigger)

**File:** `frontend/src/features/[feature]/components/[component].jsx`

**Pattern:**
```javascript
"use client"
import { useForm } from "react-hook-form"
import { zodResolver } from "@hookform/resolvers/zod"
import { [featureName]Schema } from "@/lib/validations/auth-schemas"

export function FeatureContent({
  isSubmitting,    // from hook via parent
  onSave,          // from hook via parent
}) {
  const {
    register,
    handleSubmit,
    formState: { errors },
    reset,
  } = useForm({
    resolver: zodResolver([featureName]Schema),
    mode: "onTouched",
    reValidateMode: "onChange",
    defaultValues: {
      fieldA: "",
      fieldB: "",
    },
  });

  return (
    <form onSubmit={handleSubmit((data) => onSave(data, reset))}>
      <Input {...register("fieldA")} />
      {errors.fieldA && <p>{errors.fieldA.message}</p>}

      <Button type="submit" disabled={isSubmitting}>
        {isSubmitting ? "Saving..." : "Save"}
      </Button>
    </form>
  );
}
```

**Rules:**
- Component is DUMB — receives `isSubmitting` and `onSave` via props only
- `useForm` lives INSIDE the component (form UI state is local)
- Show/hide toggles, local UI booleans live INSIDE the component
- `defaultValues` keys MUST match Zod schema keys exactly
- Submit calls `onSave(data, reset)` — passes form data AND reset function
- Never import `useDispatch`, `useSelector`, API clients, or `notify` here

**Boundary contract OUT:**
- Calls `onSave(data, reset)` where `data` is the validated form object
- `reset` is react-hook-form's reset function

### STAGE 4: Hook (Business Logic Orchestrator)

**File:** `frontend/src/features/[feature]/hooks/use[Feature].js`

**Pattern:**
```javascript
import { useState } from "react";
import { useDispatch } from "react-redux";
import { featureThunk } from "@/store/slices/[domain]/[domain]-thunks";
import { notify } from "@/lib/notify";

export function useFeature() {
  const dispatch = useDispatch();
  const [isSubmitting, setIsSubmitting] = useState(false);

  const handleSave = async (formData, resetForm) => {
    setIsSubmitting(true);
    try {
      await dispatch(featureThunk(formData)).unwrap();
      notify.success("Action completed successfully");
      if (resetForm) resetForm();
    } catch (error) {
      if (!error?.isGlobalError) {
        notify.error(error?.message || "Action failed");
      }
    } finally {
      setIsSubmitting(false);
    }
  };

  return { isSubmitting, handleSave };
}
```

**Rules:**
- Hook owns: dispatch, notify, isSubmitting state
- Hook does NOT own: form field state, show/hide toggles (those stay in component)
- Always uses `.unwrap()` on thunk dispatch to get resolved/rejected value
- Always checks `!error?.isGlobalError` before showing error toast
  (interceptor already showed toast for 403, 429, 500, network errors)
- Notify is imported from `@/lib/notify` (Rule F4) — NEVER from "sonner"
- Success notification ONLY here — NOT in the thunk (prevents duplicates)

**Boundary contract IN:** `handleSave(formData, resetForm)` — positional args from component

**Boundary contract OUT:** `dispatch(thunk(formData)).unwrap()` — passes raw formData to thunk

**The `isGlobalError` guard explained:**
```
Backend returns 400 → interceptor passes through → isGlobalError: false → hook shows toast
Backend returns 403 → interceptor shows toast + sets isGlobalError: true → hook skips toast
Backend returns 429 → interceptor shows toast + sets isGlobalError: true → hook skips toast
Backend returns 500 → interceptor shows toast + sets isGlobalError: true → hook skips toast
Network error     → interceptor shows toast + sets isGlobalError: true → hook skips toast
```

### STAGE 5: Redux Thunk (Async Action)

**File:** `frontend/src/store/slices/[domain]/[domain]-thunks.js`

**Pattern:**
```javascript
import { domainService } from "@/services/domain/[domain]-service";
import { createAppThunk } from "@/store/utils/thunk-utils";

export const featureAction = createAppThunk(
  "[domain]/[actionName]",
  async (payload, { signal }) => {
    const response = await domainService.featureMethod(payload, { signal });
    return response.data;
  },
  "Default error message for this action"
);
```

**Rules:**
- Uses `createAppThunk` wrapper (not raw `createAsyncThunk`)
- Action type string format: `"[domain]/[actionName]"` (e.g., `"user/changePassword"`)
- Passes `{ signal }` for request cancellation support
- Returns `response.data` (normalized response)
- Does NOT dispatch `showNotification` — hook handles success toast
- Does NOT catch errors — `createAppThunk` handles rejection and wraps errors
- The `createAppThunk` wrapper produces this error shape on rejection:
  ```javascript
  {
    message: string,
    errorCode: string,
    status: number,
    isGlobalError: boolean
  }
  ```

**Boundary contract IN:** `payload` — raw object from hook (identical to form data)

**Boundary contract OUT:** `domainService.method(payload, { signal })` — passes to service

### STAGE 6: Service Layer (API Abstraction)

**File:** `frontend/src/services/domain/[domain]-service.js`

**Pattern:**
```javascript
async featureMethod(data, config = {}) {
  this.validateUserData(data, "featureKey");

  const response = await privateClient.post(
    domainEndpoints.FEATURE_ENDPOINT,
    data,
    config
  );
  return normalizeResponse(response);

```

**Rules:**
- `validateUserData` checks required fields are present before hitting network

  ```javascript
  const validations = {
    featureKey: ["fieldA", "fieldB"],  // must match Zod schema field names
  };
  ```
- HTTP method must match backend route (GET/POST/PATCH/PUT/DELETE)
- Payload passed as-is — no field renaming or stripping
- `normalizeResponse` standardizes the axios response shape

**Available clients:**
| Client | Use Case |
|--------|----------|
| `privateClient` | Authenticated requests (has auth interceptor) |
| `publicClient` | Unauthenticated requests (login, register, forgot-password) |

**Boundary contract IN:** `data` — raw object from thunk

**Boundary contract OUT:** `privateClient.[method](endpoint, data)` — HTTP request

### STAGE 7: Endpoint Definition (URL Mapping)

**File:** `frontend/src/services/api/endpoints/[domain]-endpoints.js`

**Pattern:**
```javascript
class DomainEndpoints {
  constructor() {
    this.PREFIX = "/[route-prefix]";  // must match backend app.use() mount
  }

  get FEATURE_ENDPOINT() {
    return `${this.PREFIX}/path/to/resource`;
  }
}
```

**Rules:**
- PREFIX must match the backend mount path in `app.js`
- Backend mounts user routes at: `app.use("/api/v1/user", userRoutes)` → PREFIX = `"/user"`
- Backend mounts auth routes at: `app.use("/api/v1/auth", authRoutes)` → PREFIX = `"/auth"`
- The axios base URL is `http://localhost:4000/api/v1` — so endpoints start AFTER `/api/v1`
- Full URL = `baseURL + PREFIX + path` = `http://localhost:4000/api/v1/user/security/password`

**Critical check:** Frontend PREFIX must match backend `app.use()` mount path EXACTLY (singular vs plural matters: `/user` not `/users`)

### STAGE 8: Axios Client (HTTP + Interceptors)

**Files:**
- `frontend/src/services/api/client/private-client.js`
- `frontend/src/services/api/client/base-client.js`

**You do NOT modify these files for new features.** They are infrastructure.

**What they do:**
- Request interceptor: Attaches `Authorization: Bearer <token>` header
- Response interceptor processes errors in this order (Rule B7):
  1. 401 TOKEN_EXPIRED → silent refresh + retry (no toast)
  2. 401 refresh failed → toast "session expired" + clearCredentials
  3. Network error → toast with ID `global-network`
  4. 403 → toast with ID `global-403`
  5. 429 → toast with ID `global-429`
  6. 500 → toast with ID `global-500`
  7. 502/503/504 → toast with ID `global-5xx`
  8. All other errors (including 400) → pass through (no toast)

**Why this matters for hooks:**
- 400 errors reach the hook's catch block with `isGlobalError: false`
- 403/429/500/network errors reach with `isGlobalError: true`
- The hook checks `!error?.isGlobalError` to avoid duplicate toasts

### STAGE 9: Backend Router (Route Registration)

**File:** `backend/routes/[domain]/[domain]-routes.js`

**Pattern:**
```javascript
import express from "express";
import { authTokenMiddleware } from "../../middleware/auth/authTokenMiddleware.js";
import { featureLimiter } from "../../middleware/security/rate-limiters.js";
import { featureValidationRules } from "../../validators/validationRules.js";
import { handleValidationErrors } from "../../middleware/validation/validation-middleware.js";
import { handleFeature } from "../../controllers/[domain]/feature.controller.js";

const router = express.Router();

// METHOD /api/v1/[domain]/path
router.post(
  "/path",
  featureLimiter,              // 1. Rate limit
  authTokenMiddleware,          // 2. Auth (if protected)
  featureValidationRules,       // 3. Validation rules
  handleValidationErrors,       // 4. Validation error formatter
  handleFeature                 // 5. Controller
);

export default router;
```

**Middleware chain order (ALWAYS in this sequence for protected routes):**
1. Rate limiter — blocks before any work
2. Auth middleware — verifies JWT, sets `req.user`
3. Validation rules — checks req.body fields
4. Validation error handler — formats errors into standard envelope
5. Controller — handles request

**For public routes** (login, register, forgot-password): skip step 2.

**Route is mounted in `backend/app.js`:**
```javascript
app.use("/api/v1/user", userRoutes);
app.use("/api/v1/auth", authRoutes);
```

### STAGE 11: Auth Middleware (Token Verification)

**File:** `backend/middleware/auth/authTokenMiddleware.js`

**You do NOT modify this file for new features.** It is infrastructure.

**What it does:**
- Reads token from `Authorization` header (`Bearer <token>`)
- Verifies JWT signature and expiry
- Sets `req.user = decoded.UserInfo`
- Key property: `req.user.userId` — this is the authenticated user's ID

**What it returns on failure:**
| Scenario | Status | errorCode |
|----------|--------|-----------|
| Missing/malformed header | 401 | NO_ACCESS_TOKEN |
| Expired token | 401 | TOKEN_EXPIRED |
| Invalid/revoked token | 401 | TOKEN_INVALID |

### STAGE 12: Backend Validation Rules (express-validator)

**File:** `backend/validators/validationRules.js`

**Pattern:**
```javascript
export const featureValidationRules = [
  body("fieldA")
    .notEmpty()
    .withMessage("Field A is required").bail()
    .isString()
    .withMessage("Field A must be a string").bail()
    .isLength({ max: 128 })
    .withMessage("Field A length exceeded limit"),

  passwordRules("passwordField"),  // shared helper for password strength

  body("confirmField")
    .notEmpty()
    .withMessage("Confirm field is required").bail()
    .custom((value, { req }) => {
      if (value !== req.body.passwordField) {
        throw new Error("Fields must match");
      }
      return true;
    }),
];
```

**Rules (Rule V2):**
- `.bail()` after every `.notEmpty()` and `.isString()`
- No `.escape()` anywhere (XSS handled at output layer)
- Use `passwordRules(fieldName)` shared helper for new password fields
- Login password: validate existence only, NOT strength
- Error messages are user-facing, not developer-facing

**Shared helpers available:**
| Helper | Use For |
|--------|---------|
| `passwordRules(field)` | Any new password field (signup, reset, change) |
| `emailRules({ checkDisposable })` | Any email field |

**Critical:** Field names in validation rules MUST match what the frontend sends in the request body.

### STAGE 14: Controller (Thin HTTP Adapter)

**File:** `backend/controllers/[domain]/[feature].controller.js`

**Pattern:**
```javascript
import { featureUseCase } from "../../use-cases/[domain]/[feature].use-case.js";
import { sendUseCaseResponse } from "../auth/auth-shared.js";

export const handleFeature = async (req, res, next) => {
  try {
    const { fieldA, fieldB } = req.body;
    const userId = req.user.userId;

    const result = await featureUseCase({
      userId,
      fieldA,
      fieldB,
```

**Rules (Rule B9):**
- Controller is THIN — extracts from req, calls use case, passes to `sendUseCaseResponse`
- No business logic in controller
- Extract only what the use case needs from `req.body` (skip confirmPassword, etc.)
- User ID comes from `req.user.userId` (set by auth middleware)
- Always wrap in try/catch, forward to `next(error)`
- Import `sendUseCaseResponse` from `"../auth/auth-shared.js"` (path is relative to controllers dir)

**Boundary contract IN:** `req.body` (validated), `req.user.userId` (authenticated)

**Boundary contract OUT:** Use case receives a plain object, NOT the raw req

### STAGE 15: Use Case (Core Business Logic)

**File:** `backend/use-cases/[domain]/[feature].use-case.js`

**Pattern:**
```javascript
import Model from "../../model/Model.js";
import { hashPassword, comparePassword } from "../../utilities/auth/hash-utils.js";
import logger from "../../utilities/general/logger.js";

export async function featureUseCase({ userId, fieldA, fieldB }) {
  try {
    // 1. Find entity
    const entity = await Model.findById(userId).select("+hiddenField");

    if (!entity) {
      return {
        success: false,
        statusCode: 404,
        message: "Not found",
        errorCode: "NOT_FOUND",
      };
    }

    // 2. Business logic checks
    // 3. Mutations
    // 4. Save

    await entity.save();

    logger.info({ userId }, "Action completed");

    return {
      success: true,
      statusCode: 200,
      message: "Action completed successfully",
      data: { /* optional response data */ },
    };
  } catch (error) {
    logger.error({ err: error, userId }, "Use case error");

    if (error.name === "MongoError" || error.name === "MongoServerError") {
      return {
        success: false,
        statusCode: 503,
        message: "Service temporarily unavailable",
        errorCode: "DATABASE_ERROR",
      };
    }

    return {
      success: false,
      statusCode: 500,
      message: "Action failed",
      errorCode: "ACTION_FAILED",
    };
  }
}
```

**Result object shape (ALWAYS this structure):**
```javascript
{
  success: boolean,
  statusCode: number,
  message: string,
  errorCode?: string,    // present on failure
  data?: object,         // present on success if returning data
}
```

**Rules:**
- Use case is PURE business logic — no req, no res, no HTTP concepts
- Receives a plain DTO object, returns a plain result object
- Status codes: use 400 for user errors (NOT 401 for auth'd users — avoids interceptor confusion)
- All errors caught internally — never throws to controller
- Uses `select("+field")` for hidden fields (password has `select: false` on model)
- Error codes are UPPER_SNAKE_CASE

**Available utility functions:**
| Function | Import From | Purpose |
|----------|-------------|---------|
| `hashPassword(plain)` | `../../utilities/auth/hash-utils.js` | Bcrypt hash |
| `comparePassword(plain, hash)` | `../../utilities/auth/hash-utils.js` | Bcrypt compare |
| `generateToken()` | `../../utilities/auth/crypto-utils.js` | Random tokens |
| `hashToken(token)` | `../../utilities/auth/crypto-utils.js` | SHA-256 hash |

### STAGE 17: sendUseCaseResponse (Response Formatting)

```javascript
export const sendUseCaseResponse = (req, res, result) => {
  return apiResponseManager(req, res, {

    success: result.success,
    message: result.message,
    ...(result.errorCode && { errorCode: result.errorCode }),
```

```javascript
  data: { ... },       // if provided

  requestId: "..."
}
```

Error:
```javascript
{
  success: false,
  message: "...",
  errorCode: "...",
  timestamp: "ISO",
  requestId: "..."
}
```

### STAGES 18-22: Response Return Path

**These are infrastructure. You do NOT modify them for new features.**

**Stage 18 — Axios Interceptor:**
- 400 errors pass through with `isGlobalError: false`
- 403/429/500/network → toast shown, `isGlobalError: true`

**Stage 19 — createAppThunk wrapper:**
- Catches errors, normalizes to: `{ message, errorCode, status, isGlobalError }`
- This is what the hook's `catch(error)` receives

**Stage 20 — Hook catch/then:**
- Success: `notify.success()` + `resetForm()`
- Error: check `!error?.isGlobalError`, then `notify.error(error?.message)`

**Stage 21 — Redux Slice:**
- `pending`: `isLoading = true, error = null`
- `fulfilled`: `isLoading = false, error = null` + any state updates
- `rejected`: `isLoading = false, error = action.payload`

**Stage 22 — Notification:**
- Success toast: ONLY in hook (not in thunk)
- Error toast for 400s: ONLY in hook
- Error toast for 403/429/500/network: ONLY in interceptor (hook skips via isGlobalError)

## ATOMIC TASK LIST TEMPLATE

For any new feature, execute these tasks in this exact order.
Each task targets exactly one file.

### Backend Tasks (execute first)

```

  Inputs: field names, types, constraints
  Verify: Field names match frontend Zod schema field names

TASK B3: Create use case
  File: backend/use-cases/[domain]/[feature].use-case.js
  Action: Create new file with business logic
  Inputs: DTO shape from controller

  Verify: Uses correct utility imports, handles all error cases

TASK B4: Create controller
  File: backend/controllers/[domain]/[feature].controller.js
  Action: Create thin controller
  Inputs: req.body fields, req.user.userId

TASK B5: Register route
  File: backend/routes/[domain]/[domain]-routes.js

  Chain: limiter → authTokenMiddleware → validationRules → handleValidationErrors → controller
```

### Frontend State Tasks (execute second)
